# 知维图谱：基于大语言模型的概念知识图谱构建工具

> **摘要**：知维图谱是一款面向中小学教师的桌面端教育应用。系统以概念转变理论和有意义学习理论为基础，利用大语言模型的语义理解能力，辅助教师在备课时梳理学科概念体系、预判新概念引入可能引发的认知冲突与重构需求，并通过智能体对话实现概念图谱的半自动化构建。系统基于 Electron + React + D3.js 实现，数据本地存储，打包为 Windows 桌面应用分发。

**关键词**：概念图谱；大语言模型；教师备课工具；概念转变理论；认知冲突

---

## 一、所解决的教育痛点和问题

### 1.1 教师概念体系梳理缺乏系统工具

教师在备课时需要完成一项关键的认知任务：梳理本单元的概念及其相互关系。以初中物理"力学"为例，讲授"加速度"之前，教师需明确学生已掌握哪些先备知识（位置、速度、时间等），这些概念间存在怎样的前提、类比或上下位关系，以及新概念可能引发哪些认知冲突（Barron & Darling-Hammond, 2008）。Shulman（1986）提出的"学科教学知识"（PCK）理论指出，优秀教师区别于学科专家的关键，在于能将学科知识转化为适合学生理解的表征形式。然而，概念体系的梳理在教师培训和教学实践中几乎完全依赖个体经验和直觉，缺乏系统化工具支撑。

### 1.2 迷思概念识别的滞后性

教育心理学研究表明，学生在正式学习前已通过日常经验形成了大量"前概念"，其中与科学概念存在偏差的部分被称为"迷思概念"（Vosniadou, 1994）。例如，许多学生持有"力维持运动"的直觉信念，学习加速度时又将"加速度"等同于日常语义的"加速"（diSessa, 1993）。传统教学通常在考试或作业后才"发现"这些迷思概念，此时教学已完成，只能补救。迷思概念一旦形成具有较强的认知顽固性，事后纠正的成本远高于事前预防。

### 1.3 教育 AI 中教师角色的边缘化

当前大量 AI 教育产品将 AI 定位为"替代教师的教学者"——AI 讲课、出题、批改，教师逐渐被边缘化（Selwyn, 2019）。这不仅与教育学基本规律相悖，也引发了教师群体对技术的抵触。教育 AI 的正确方向应是"辅助教师更好地教"——面向教师的备课环节，而非"替代教师去教学生"。知维图谱正是基于这一理念设计的。

## 二、现有应用的现状调研

概念图工具方面，CmapTools 和 XMind 是最具代表性的产品。前者由 IHMC 开发，支持多人在线协作构建概念图；后者内置丰富的图形模板，导出格式多样。然而，这些工具的共性是"画图工具"——提供的是图形编辑能力而非知识分析能力，概念关系类型和体系完整性的判断完全依赖用户，对时间有限的教师反而增加额外负担。

AI 教学平台方面，清华大学的 OpenMAIC（于济凡 等, 2025）利用多智能体架构实现了 AI 教师、助教和同学的协同课堂，在清华大学真实课堂中进行了大规模测试。但其核心定位是"AI 驱动的教学交付系统"——关注"AI 怎么讲"，而非"学生怎么想"。商业产品如 Khan Academy 的 Khanmigo、国内科大讯飞的智能批改系统等，则主要聚焦于自适应学习和自动化评测，缺乏面向教师概念教学设计的工具。

知识图谱在教育中虽有探索（如 KnowEdu 项目的学科知识图谱构建）（Chen et al., 2018），但多由专家手工构建，且描述的是"客观的知识体系"而非"学生头脑中的认知结构"。这种从"应然"到"实然"的视角转换——考虑学生的概念掌握水平、前概念偏差和迷思概念分布——是当前几乎所有产品都未触及的空白。

## 三、关键技术和理论基础

### 3.1 理论框架

皮亚杰（Piaget, 1952）的认知发展理论提出了同化（将新信息纳入已有图式）和顺应（重组已有图式以适应新信息）两种认知机制。教师在引入新概念时需要判断：这个新知识是通过"同化"平滑整合，还是需要引发"顺应"式的认知重构。

概念转变理论（Posner et al., 1982）进一步指出，真正的科学学习往往需要满足"不满、可理解、合理、丰富"四个条件。Chi（1992）的"本体论错位"理论则解释了为何某些概念转变特别困难——学生将概念归属到了错误的本体论类别。

奥苏贝尔（Ausubel, 1968）的有意义学习理论强调"影响学习最重要的因素是学习者已经知道什么"，新知识需要与已有认知结构建立实质性的、非任意的联系。变易理论（Marton & Booth, 1997）则指出，学习者必须通过"对比"来辨识概念的关键特征。

基于上述理论，知维图谱将新概念引入的分析维度设计为：（1）同化锚点——可作为新知识基础的已有概念及其类比桥梁；（2）认知冲突——已有概念体系中与新概念矛盾的内容及可能的迷思概念；（3）认知重构——需要发生根本性概念转变的节点；（4）难度预测与教学建议。

### 3.2 技术架构

系统采用 Electron + React + TypeScript + D3.js 技术栈。Electron 主进程负责本地 JSON 文件读写和 LLM API 调用，渲染进程负责 UI 展示和用户交互。所有项目数据存储在 `Documents/ConceptEcology/projects/` 目录下，无需云数据库。

**概念生态模拟**：教师输入新概念后，系统将已有概念体系摘要发送给 LLM。LLM 通过 System Prompt 被赋予"认知心理学家"角色（内置皮亚杰、Posner 等理论框架），返回结构化的 JSON 分析结果。

**智能体对话交互（"小知"助手）**：采用自实现的工具调用机制——将知识库操作工具（增删改概念、建立关系）的描述嵌入 System Prompt，LLM 通过输出 `<tool>` XML 标签触发工具调用，后端解析后直接执行本地文件操作并返回结果。

**D3.js 力导向图**：概念节点和关系边通过力模拟算法渲染，颜色编码帮助教师快速定位：绿色=同化锚点、红色=认知冲突、蓝色=激活概念、紫色=新概念。

**桌面端本地化**：数据完全存储在用户本地，消除隐私隐患，打包为 Windows 便携文件夹后可直接分发使用。

## 四、不足与展望

### 4.1 当前局限

LLM 分析质量高度依赖模型推理能力，不同模型的分析深度和准确性差异显著，且存在"幻觉"风险，系统目前缺乏对 LLM 输出的后置校验。当前的 Prompt 设计主要面向理科中概念层次分明的知识体系，对文史类学科中边界模糊的概念结构验证不足。系统对认知冲突和难度的预测完全基于 LLM 的先验知识推理，尚未接入真实学生数据进行验证校准。此外，概念图谱的初始构建需要教师投入前置工作量，可能构成使用门槛。

### 4.2 未来方向

一是引入学科领域自适应机制，根据不同学科自动调整分析策略。二是在授权前提下接入学生诊断性测试和概念图作业等数据，将 LLM 预测与学生实际表现进行对照，逐步将系统从"推测工具"升级为"实证决策支持系统"。三是扩展为支持同校教师团队共享图谱的协作教研平台，形成学校层面的知识资产积累。四是与主流学习管理系统（如 Moodle）对接，真正融入教师的日常工作流。五是探索语音输入和手绘草图等多模态交互方式，降低使用门槛。
